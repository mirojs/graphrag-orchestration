## Why the committed code is still wrong

### Problem 1 — RELATED_TO degree filter is backwards

Concept nodes ("Agreement", "Owner", "Builder", "warranty") have the **highest**
RELATED_TO degree because every document annotates generic roles with `RELATED_TO`
edges. Contracting principals with specific names (e.g. "Contoso Lifts LLC", which
appears only in one purchase contract) have **low** degree.

Result: concept nodes pass the `>= 2` filter; specific contractors fail it.

Benchmark evidence — entity_doc_map returned 19 entities including:
```
Agreement, Owner, 2010-06-15, property, service contract, Agent, Builder,
Authorized Representative, Hawaii, County, municipality, warranty
```
…while "Contoso Lifts LLC" (a legitimate contracting party) was **missing**.

Score after entity-doc map: **exact=0.67, f1=0.52, containment=0.82** — still not 3/3.

### Problem 2 — PPR top-20 scope limitation

`entity_scores[:20]` is the PPR ranked list seeded from surviving_triples for the
specific query. For Q-G6 (which spans ALL documents), Contoso Lifts LLC did not
appear in the top 20 PPR-ranked entities.

Even a correct filter could not surface it because it was never in the input list.

### Problem 3 — hardcoded query vocabulary (corpus-specific)

`_is_exhaustive_entity_query()` matches only:
  `(list|enumerate|identify|name) … all … (parties|organizations|entities|named parties)`

This does **not** generalise to:
- "What are all the contract amounts mentioned across documents?"
- "List all dates and deadlines in the documents"
- "Which locations appear in each document?"

The user's core objection applies: this is still document/domain-specific code.

### Problem 4 — hardcoded response template (corpus-specific)

```python
response_text = (
    "The following named parties/organizations appear across the documents:\n\n"
    + "\n".join(lines)
)
```

This string is written for the parties/contracts domain and would be wrong for any
other exhaustive query type.

## Root cause (unchanged — §33.25)

The fundamental architectural gap is **triple discard before synthesis**:

```
surviving_triples  (LLM-confirmed, structurally rich)
        ↓
  used only as PPR seed IDs
        ↓  ← discarded here
synthesis receives:  (entity_name, ppr_score)  +  raw text chunks only
```

For exhaustive questions, synthesis reads the raw text and enumerates **every**
named entity in the text, not just those that are structurally significant. The
`graph_structural_header` (committed) prepends surviving_triples to the synthesis
context but cannot override a sufficiently text-heavy context window.

## What needs to be solved (open questions)

### Q1 — What is an "exhaustive" query, generically?

An exhaustive query is one where the answer must be **complete across the corpus**,
not merely **relevant to the question**. Examples:
- "List all X in each document" (enumeration)
- "Which documents mention X?" (coverage)
- "Find all occurrences of X across all documents" (scan)

PPR+synthesis is optimised for **relevance** not **completeness**. This tension is
the core architectural gap.

### Q2 — Can we detect exhaustive intent without hardcoding vocabulary?

Pattern signals for exhaustive intent (vocabulary-independent):
- Quantifier: "all", "every", "each", "across all", "in each"
- Scope: "across the documents", "in all documents", "throughout"
- Task verb: "list", "enumerate", "identify", "find all", "what are all"

A detector based on these structural signals (no domain nouns) would generalise.

### Q3 — Given exhaustive intent is detected, what retrieval strategy is correct?

Options:

**A) Graph-direct query** — skip PPR entirely, hit the MENTIONS index with a
query-derived entity-type mapping. Requires: an LLM sub-call to map query object
("parties", "locations", "amounts") to schema type (ORGANIZATION, LOCATION, …).
Still fails for things not stored as entity nodes (amounts, dates).

**B) Synthesis-constraint mode** — keep PPR+synthesis but pass exhaustive intent
as a hard synthesis constraint: "return ONLY entities present in the Graph
Structural Evidence; do not add entities from raw text." Requires: the structural
evidence must be complete (full entity-doc table, not just surviving_triples).

**C) Two-pass synthesis** — first pass: extract all entity candidates from each
document chunk with a focused extraction prompt. Second pass: deduplicate and
describe. Expensive but completeness-preserving.

**D) MENTIONS index + LLM re-rank** — query ALL entities of every type from the
MENTIONS index, pass the full list to the LLM to filter to the ones the query
actually asks about. General, assumes only that the KG was indexed correctly.

## Key facts to remember

- Entity nodes in Neo4j have a `type` property: `ORGANIZATION`, `PERSON`,
  `DOCUMENT`, `LOCATION`, `CONCEPT` (default). Indexed via `entity_type` index.
- The `importance_score` property (`degree * 0.3 + chunk_count * 0.7`) and
  `chunk_count` property are computed and stored on Entity nodes — these could be
  used as filters instead of a live RELATED_TO degree count.
- `chunk_count` = number of text chunks that MENTION the entity. This is a better
  proxy for "how substantively is this entity featured in documents" than RELATED_TO
  degree. A contracting principal will have `chunk_count > 1`; a passing mention
  (jurisdiction, job-site name) may have `chunk_count = 1`.
- PPR entity_scores[:20] is the retrieval scope for Step 4.5. For cross-corpus
  exhaustive queries this is too narrow — it biases toward whichever document
  the query's seed entities are most connected to.
